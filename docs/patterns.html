Architectural Patterns

Clean Architecture (Hexagonal / Ports & Adapters)

Backend microservice project structure

Domain entities (Ticket.cs, TicketResponse.cs) are isolated from infrastructure concerns. EF Core DbContext implementations and ASP.NET controllers sit at the outer layers, allowing the database provider to be swapped (SQLite ↔ SQL Server) without modifying business logic.

API Gateway Pattern

src/ResolveDesk.Gateway/ — YARP Reverse Proxy

Provides a single network entry point (localhost:5100) for the frontend. Routes are declaratively configured in appsettings.json. This abstracts internal service addresses, simplifies CORS, and centralizes cross-cutting concerns like logging.

// The Angular frontend only knows ONE URL:
private apiUrl = 'http://localhost:5100/api/tickets';
// YARP transparently routes to http://localhost:5103/api/tickets

Behavioral Patterns

Dependency Injection (DI)

Program.cs in each service + Constructor injection in controllers

ASP.NET Core's built-in IoC container resolves all dependencies: DbContext, IPublishEndpoint (MassTransit), UserManager<T> (Identity), and ILogger<T>. No manual object creation — everything is wired at startup.

// TicketsController.cs — Constructor injection
public TicketsController(
    TicketDbContext context,           // EF Core database context
    IPublishEndpoint publishEndpoint,  // MassTransit event publisher
    ILogger<TicketsController> logger  // Structured logging
) { ... }

Publish-Subscribe (Pub/Sub)

MassTransit InMemory Bus — Consumers/ folder in TicketCore

Event producers (TicketsController) publish messages (ITicketCreatedEvent, ITicketStatusChangedEvent) without knowing who consumes them. MassTransit routes events to registered consumers (TicketCreatedConsumer, TicketStatusChangedConsumer) automatically.

// Producer: TicketsController.cs
await _publishEndpoint.Publish<ITicketCreatedEvent>(new { ... });

// Consumer: TicketCreatedConsumer.cs — Decoupled handler
public class TicketCreatedConsumer : IConsumer<ITicketCreatedEvent> {
    public async Task Consume(ConsumeContext<ITicketCreatedEvent> context) {
        _logger.LogInformation($"Ticket created: {context.Message.TicketId}");
    }
}

Repository Pattern (via EF Core)

TicketDbContext acts as the Unit of Work + Repository

EF Core's DbContext naturally implements the Repository pattern through DbSet<T> collections. Controllers query via LINQ (_context.Tickets.Where(...)) and persist changes with SaveChangesAsync().

// Repository-like access via EF Core DbSet
var tickets = await _context.Tickets
    .Include(t => t.Responses)    // Eager load navigation property
    .Where(t => t.CustomerId == userId)
    .OrderByDescending(t => t.CreatedAt)
    .ToListAsync();

Frontend Patterns

Reactive State Store (Angular Signals)

auth.service.ts — currentUser = signal<User | null>(null)

Instead of RxJS BehaviorSubjects or NgRx stores, the app uses Angular 17's native Signals for state management. The currentUser signal automatically triggers change detection in consuming components when updated — no Zone.js required, no manual subscription cleanup.

// auth.service.ts — Signal-based state
currentUser = signal<User | null>(null);

// On login success:
this.currentUser.set(user); // All dependent UI updates automatically

// In dashboard.component.ts:
this.currentUser = this.authService.currentUser(); // Read signal value

Intercepting Filter (HTTP Interceptor)

src/ResolveDesk.UI/src/app/interceptors/auth.interceptor.ts

A functional HTTP interceptor (HttpInterceptorFn) automatically attaches the JWT Bearer token to every outbound API request. This eliminates the need to manually add authorization headers in every service call.

// auth.interceptor.ts — Automatic token attachment
export const authInterceptor: HttpInterceptorFn = (req, next) => {
    const user = inject(AuthService).currentUser();
    if (user?.token) {
        req = req.clone({
            setHeaders: { Authorization: `Bearer ${user.token}` }
        });
    }
    return next(req);
};

// Registered in app.config.ts:
provideHttpClient(withInterceptors([authInterceptor]))

Route Guards

src/ResolveDesk.UI/src/app/guards/auth.guard.ts

A functional route guard protects the /dashboard route. If the user is not authenticated, they are redirected to /login. This is a client-side check complementing the server-side JWT validation.

// app.routes.ts — Protected route
{ path: 'dashboard', component: DashboardComponent, canActivate: [authGuard] }
