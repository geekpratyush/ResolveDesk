Architecture & Infrastructure

Deep dive into the microservices topology, database strategy, YARP gateway, MassTransit event bus, and JWT authentication pipeline.

YARP API Gateway

The Gateway is a lightweight ASP.NET Core application using YARP (Yet Another Reverse Proxy) to route all frontend requests to the correct backend microservice.

graph LR
    Client["Angular UI :4300"] -->|"All API calls to :5100"| GW["YARP Gateway :5100"]
    GW -->|"/api/auth/*"| ID["Identity Service :5101"]
    GW -->|"/api/tickets/*"| TC["Ticket Core :5103"]

YARP Route Configuration

// src/ResolveDesk.Gateway/appsettings.json { "ReverseProxy": { "Routes": { "auth-route": { "ClusterId": "identity-cluster", "Match": { "Path": "/api/auth/{**catch-all}" } }, "tickets-route": { "ClusterId": "tickets-cluster", "Match": { "Path": "/api/tickets/{**catch-all}" } } }, "Clusters": { "identity-cluster": { "Address": "http://localhost:5101" }, "tickets-cluster": { "Address": "http://localhost:5103" } } } }
Why YARP? It allows a single CORS-friendly URL for the Angular frontend while abstracting the multi-port backend layout. The Angular services only know about localhost:5100.

Database Strategy (EF Core Dual Provider)

Both microservices use Entity Framework Core 8 with a dual-provider pattern: SQLite for zero-install local demos, SQL Server for production environments.

graph TD
    subgraph Identity Service
        IDP["Program.cs"] -->|"Connection string check"| DEC{"Contains 'Data Source='?"}
        DEC -->|Yes| SQLITE1["UseSqlite(identity.db)"]
        DEC -->|No| SQLSRV1["UseSqlServer(connStr)"]
    end
    subgraph Ticket Core Service
        TCP["Program.cs"] -->|"Connection string check"| DEC2{"Contains 'Data Source='?"}
        DEC2 -->|Yes| SQLITE2["UseSqlite(tickets.db)"]
        DEC2 -->|No| SQLSRV2["UseSqlServer(connStr)"]
    end

Dual Provider Detection Code

// Program.cs — Dual DB provider setup var connectionString = builder.Configuration.GetConnectionString("DefaultConnection"); var isSqlite = connectionString.Contains("Data Source="); if (isSqlite) { options.UseSqlite(connectionString); // Zero-install local mode } else { options.UseSqlServer(connectionString); // Production/staging }
Auto-Seeding: On startup, if the database is empty, Program.cs seeds demo users (Admin, Support, Customer) and 8 sample tickets with pre-written conversation threads. The SQLite .db files are created automatically in the project root.

MassTransit InMemory Event Bus

MassTransit is a .NET message bus framework. In this application, it runs with the InMemory transport — no RabbitMQ, no Docker, no external infrastructure needed.

sequenceDiagram
    autonumber
    participant Ctrl as TicketsController
    participant Bus as MassTransit Bus (InMemory)
    participant C1 as TicketCreatedConsumer
    participant C2 as TicketStatusChangedConsumer

    Ctrl->>Bus: Publish ITicketCreatedEvent
    Bus-->>C1: Deliver (in-process)
    C1->>C1: Log notification simulation

    Ctrl->>Bus: Publish ITicketStatusChangedEvent
    Bus-->>C2: Deliver (in-process)
    C2->>C2: Log status change notification

MassTransit Registration

// Program.cs — MassTransit InMemory configuration builder.Services.AddMassTransit(x => { x.AddConsumer<TicketCreatedConsumer>(); x.AddConsumer<TicketStatusChangedConsumer>(); x.UsingInMemory((context, cfg) => { cfg.ConfigureEndpoints(context); // Auto-register consumer endpoints }); });

Event Interfaces (Shared Contract)

// src/ResolveDesk.Shared/Events/ITicketCreatedEvent.cs public interface ITicketCreatedEvent { Guid TicketId { get; } string CustomerId { get; } string Title { get; } string CustomerEmail { get; } DateTime CreatedAt { get; } }
InMemory Limitation: Messages are delivered in-process and are volatile — if the service restarts, unprocessed messages are lost. This is ideal for local demos but should be replaced with RabbitMQ or Azure Service Bus in production.

JWT Authentication Pipeline

Authentication uses HS256-signed JWT Bearer tokens. The token carries user identity claims (sub, email, name, roles) that the backend validates on every request.

sequenceDiagram
    autonumber
    actor User
    participant UI as Angular UI
    participant INT as auth.interceptor.ts
    participant GW as YARP Gateway
    participant API as Backend Service
    
    User->>UI: Logs in, receives JWT
    UI->>UI: Stores token in localStorage
    UI->>INT: HTTP request intercepted
    INT->>INT: Clones request, adds Authorization: Bearer {token}
    INT->>GW: Sends authenticated request
    GW->>API: Proxies to backend
    API->>API: Validates JWT signature & extracts claims
    API-->>UI: Returns authorized data

Frontend Interceptor

// src/ResolveDesk.UI/src/app/interceptors/auth.interceptor.ts export const authInterceptor: HttpInterceptorFn = (req, next) => { const authService = inject(AuthService); const currentUser = authService.currentUser(); // Read Signal if (currentUser && currentUser.token) { req = req.clone({ setHeaders: { Authorization: `Bearer ${currentUser.token}` } }); } return next(req); };
